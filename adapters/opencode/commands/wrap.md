---
description: 会话收尾：更新热文件，生成开发日志，确保项目记忆完整
---

你是会话收尾助手。在用户结束工作前，执行系统性检查并生成开发日志。

## Step 1: 检查踩坑

回顾本次会话遇到的技术问题、环境问题、非显而易见的行为，逐一确认：
- 是否已记录到 `.agents/gotchas.md`？
- 未记录的立即追加（格式：`- **问题**: 解决方案/注意事项`）

## Step 2: 检查技术决策

回顾本次会话做出的技术选型、方案选择，逐一确认：
- 是否已记录到 `.agents/decisions.md`？
- 未记录的立即追加（格式：`## YYYY-MM-DD (主题)\n\n- **决策**: 原因`）

## Step 3: 更新 context.md

读取 `.agents/context.md`，检查并更新：
- "正在进行"是否反映了本次会话的变化？
- "待启动"是否需要调整？
- 如有变化，立即更新

## Step 4: 生成开发日志

回顾本次会话的完整工作内容，生成一份 devlog 文件。

**文件命名**：`docs/devlog/YYYY-MM-DD-N-简短主题.md`
- `YYYY-MM-DD` 为当天日期
- `N` 为当天的会话序号（查看 `docs/devlog/` 下同日期前缀的文件数量 +1）
- `简短主题` 用几个关键词概括本次会话的核心工作

**文件内容**：

```markdown
# YYYY-MM-DD 会话 N：简短主题

## 完成的工作
- （本次会话完成了什么，涉及已有冷存储文档的条目用一句话描述 + 链接）

## 关键变更
- （涉及的关键文件或模块变更摘要，不需要列出每个文件）
```

以下章节仅在有内容时添加，没有则省略：

```markdown
## 遇到的问题
- （开发中遇到的问题和解决方式）

## 下次继续
- （未完成的工作、后续方向）
```

**同步更新 `docs/INDEX.md`**：在"开发日志"章节追加新文件的链接。

## Step 5: 兜底检查冷存储

回顾本次会话，检查是否有以下内容产生但未落盘：

- **技术调研**：是否对比了多个方案/库？→ 未记录则写入 `docs/research/YYYY-MM-DD-主题.md`
- **模块理解**：是否深入分析了某个模块/系统的工作原理？→ 未记录则写入 `docs/knowledge/模块名.md`
- **实施计划**：是否制定了多步骤的开发计划？→ 未记录则写入 `docs/plans/YYYY-MM-DD-主题.md`

如有新增文件，同步更新 `docs/INDEX.md`。

## Step 6: 检查热文件行数

统计 `.agents/gotchas.md` 和 `.agents/decisions.md` 的行数。
如果任一文件超过 100 行，提示用户运行 `/archive`。

## 输出确认

```
会话收尾完成：

踩坑     ✓（N 条新增 / 无变化）
决策     ✓（N 条新增 / 无变化）
context  ✓（已更新 / 无变化）
devlog   ✓ docs/devlog/YYYY-MM-DD-N-主题.md
冷存储   ✓（research/knowledge/plans 新增 N 篇 / 无新增）
行数     gotchas: N 行，decisions: N 行
```
